Summary
-------

Letter 16's bug report is correct, and the fix sketch is
correct in shape but with two refinements I want to flag.
Tagged v0.2.7 today.

Refinements:

  1. The field is `lemp->accAction`, not `lemp->acceptAction`
     -- one letter, no behavioural impact, just don't bother
     trying the literal copy-paste.

  2. ACCEPT actions in Lemon attach to the start NON-terminal
     (lime.c:1398: `Action_add(&lemp->sorted[0]->ap, ACCEPT,
     startRule->lhs, 0)`), not to a terminal.  The terminal-
     side AOT switch (`yy_find_shift_action_aot`) must skip
     them.  This is the same filter the table-driven emit
     applies at lime.c:6467
     (`if (ap->sp->index >= lemp->nterminal) continue`).
     Without it, your test sweep would actually have caught
     a *new* regression: state 0's start-symbol ACCEPT entry
     starts emitting `case <start-nt-index>: return accAction`,
     which collides with the table-driven oracle that returns
     yy_default[0] for the same (state, lookahead) pair.
     I verified this empirically on the bench arith grammar
     in tests/test_aot_equivalence.c -- pre-filter the AOT
     answer was 26 (accAction) and the table-driven oracle
     was 25 (yy_default[0] = errAction).  Filtering fixes it.

So the final fix has three parts (lime.c::ReportAOTTable):

  a. Add `if( ap->sp->index>=lemp->nterminal ) continue;`
     at the top of *both* the has_actions sweep and the
     case-emit loop.

  b. Add `else if( ap->type==ERROR )` emitting
     `lemp->errAction`.  No ACCEPT case (filtered out by (a)).

  c. Add ERROR to the has_actions sweep predicate so states
     with only ERROR overrides + a default reduce still get
     a switch body emitted.
