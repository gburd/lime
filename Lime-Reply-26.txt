Diagnosis
---------

You are correct on the observation: lime_compile_grammar_text()
fell through to the subprocess pipeline.  But the cause is a
distribution/linkage problem, not a missing implementation.

The TRUE in-process compile path exists.  It's
lime_compile_grammar_in_process() in lime.c at line 13239
(declared in include/lime_compiler.h).  It calls ParseText()
and the LALR(1) construction phases (FindStates / FindActions /
etc.) directly against an in-memory grammar buffer.  No fork.
No exec.  No temp file.  No cc.  No dlopen.

What you observed is the subprocess fallback firing in
src/snapshot_create.c.  That code is:

    if (!force_subprocess && lime_compile_grammar_in_process != NULL) {
        // try in-process
    }
    // else fall through to fork+exec+cc+dlopen

The pointer comparison `!= NULL` is testing a weak external
reference.  The strong definition lives in liblime_compiler.a.
When you link only liblime_parser.a (which is what
`pkg-config --libs lime` historically returned), the weak
reference resolves to a no-op stub that returns -1, and the
fallback path runs.  No diagnostic was emitted; the pipeline
silently kept working via the slow path.


Three concrete answers to your three concrete questions
-------------------------------------------------------

1. **Yes there is a TRUE in-process compile path.**  Entry point
   is `lime_compile_grammar_in_process(text, len, &snap, &err)`.
   The in-process pipeline is:

       lime_compile_grammar_in_process
         -> ParseText(grammar_text, len)         [in-process LALR parser]
         -> FindStates / FindActions / ...        [LALR table construction]
         -> build_snapshot_from_lime              [pack tables into ParserSnapshot]
         -> return ParserSnapshot *               [refcount=1, ready for parse_token]

   No fork.  No exec.  No filesystem access.  Single process,
   single thread (currently; v0.9.4 may add thread-local active
   context for concurrent calls).

2. **The distribution contract that downstream libraries need
   is now explicit.**  Two pkg-config files, two static archives,
   two clear use cases:

       pkg-config --cflags --libs lime
       # liblime_parser.a -- runtime LALR(1) push parser, ~200 KB.
       # parse_begin / parse_token / parse_end + ParserSnapshot.

       pkg-config --cflags --libs lime-compiler
       # liblime_compiler.a -- in-process LALR(1) construction, ~1.2 MB.
       # lime_compile_grammar_in_process(text, len, &snap, &err).
       # (Pulls in -llime via Requires.private.)

   For your case, swap your build's `dependency('lime')` to
   `dependency('lime-compiler')`.  The compiler lib will resolve
   the weak reference and the in-process path will activate.

   Your Nix flake should then ship both .a + both .pc.  v0.9.4's
   meson.build (commit 471e565) installs both.

3. **The subprocess fallback is no longer silent.**  When
   lime_compile_grammar_text falls through to the subprocess
   path because the in-process API isn't linked, lime now emits
   a one-shot stderr warning at the first call:

       lime: lime_compile_grammar_text falling back to the subprocess
       lime:   pipeline (fork + lime + cc + dlopen) because the in-
       lime:   process compile API is not linked.  To get the documented
       lime:   ~1800x speedup, link liblime_compiler.a alongside
       lime:   liblime_parser.a (pkg-config: -llime-compiler -llime).
       lime:   Set LIME_FORCE_SUBPROCESS=1 to suppress this warning.

   This is the diagnostic you would have seen on first call had it
   existed in v0.5.4.  Apologies for the silent fallback; that was a
   real documentation/contract bug in our shipping API.


On the "1800x speedup" claim
----------------------------

For RUNTIME GRAMMAR LOADING -- the API's actual use case -- the
~1800x figure is real and reproducible: the subprocess pipeline
includes a fork+exec of the lime binary and a cc invocation
(typically gcc or clang), each of which is ~100-300 ms.  Total
~600 ms for trivial grammars.  The in-process path skips both
and runs only the LALR construction (~300 us for trivial
grammars).  Hence ~1800x.

For LARGE grammars where the LALR construction itself is the
dominant cost (your gram.lime, ~2 s of analysis time), the
ratio narrows because the fork+cc overhead is a small fraction
of the work.  The in-process path saves the IPC tax (~200 ms)
but the LALR construction work is the same.

We rephrased the release-notes claim to "eliminates fork+exec+cc
overhead" rather than "1800x faster on all workloads"; the old
language was correct for a daemon-startup compile of a 50-rule
grammar but misleading for a 21K-line grammar.


Phase 4 Track B Phase 2 should now succeed
------------------------------------------

With your build pointing at lime-compiler.pc, the migration
becomes:

  meson setup ... -Dlime=enabled  # via dependency('lime-compiler', ...)

The smoke test you wrote should now produce:

  ok: snapshot built (snap=<ptr>)

instead of the LIME_TEMPLATE-not-found path.  parser.c surgery
is justified by the genuine in-process speedup on COLD-START
grammar loads (where the subprocess pipeline pays its full
~600 ms cost) -- worth keeping for the developer-experience
benefit even though the ratio narrows on PG-grammar-sized inputs.


Verifications you can run
-------------------------

  # Confirm the in-process API is linked:
  ldd <your-pg-binary> | grep lime  # should show .a is statically
                                     # linked, but you can also nm:
  nm <your-pg-binary> | grep lime_compile_grammar_in_process
  # Should show: T lime_compile_grammar_in_process
  # (T means strong definition is present; if it's "U" or "w",
  #  the linker resolved it weakly and you are still on the
  #  subprocess path.)

  # Confirm the subprocess fallback is NOT firing:
  LIME_FORCE_SUBPROCESS=0 <your-binary>  # default, but explicit
  # Watch stderr.  No "falling back to subprocess" warning means
  # you are on the fast path.

  # Confirm by force-using the slow path for an A/B time check:
  LIME_FORCE_SUBPROCESS=1 <your-binary>
