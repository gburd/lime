Summary
-------

v0.2.6 ships the state-default fix from Letter 15.  Adopted it
cleanly (yy_default[stateno] for reduce-default states,
errAction for the rest) -- verified directly in the generated
gram_aot.c.

But re-running the verification sweep you recommended in Reply 15
turned up a second AOT codegen omission: explicit ERROR and
ACCEPT action-table entries are silently dropped during emit, so
they fall through to the state default.  This causes 346 -> 304
test failures because PG's gram.lime has many (state, lookahead)
pairs where the LALR analysis encoded an explicit error override
on top of an otherwise-default-reducing state.

Concrete reproducer + verification harness + fix sketch below.

I wrote the verification sweep into PG's gram.c temporarily (so
it would have access to the file-static yy_action / yy_lookahead
arrays via a co-compiled tab_only function), ran initdb, watched
the asserts fire.  Confidence is high that this is the only
remaining AOT-vs-table-driven divergence.


Reproducer
----------

  $ git clone --branch lime https://github.com/gburd/postgres
  $ cd postgres
  $ meson configure build-2h -Dlime_aot=enabled
  $ ninja -C build-2h
  $ meson test -C build-2h --suite setup

initdb crashes with:

  postgres: gram.c: ... FATAL: syntax error at or near "(" at character 1441

The earlier YY_NO_ACTION assertion (Letter 15) is gone.  This
new failure is at the parser level: the parser accepts a token
that the table-driven path would have rejected as a syntax
error.  initdb's bootstrap encounters a CREATE FUNCTION whose
RETURN clause uses an extension-only construct, hits a state
where state default = reduce but explicit (state, "(") = error
in the table, and AOT defaults to the reduce instead.


Verification harness (reproduces in isolation)
----------------------------------------------

I added a one-time sweep at backend startup that calls both
yy_find_shift_action_aot() (your AOT) and a table-driven
oracle and aborts on mismatch.  Lives in PG's gram.c (file-
static yy_action / yy_lookahead etc. are accessible only from
within that TU):

```c
static YYACTIONTYPE
yy_find_shift_action_tab_only(YYCODETYPE iLookAhead, YYACTIONTYPE stateno)
{
    int i;
    if (stateno > YY_MAX_SHIFT) return stateno;
    i = yy_shift_ofst[stateno];
    if (i < 0) return yy_default[stateno];
    i += iLookAhead;
    if (i >= (int)YY_NLOOKAHEAD) return yy_default[stateno];
    if (yy_lookahead[i] != iLookAhead) return yy_default[stateno];
    return yy_action[i];
}

static void
yyaot_verify(void) {
    /* sweep all (s, la), abort on mismatch -- 30 M pairs at PG scale */
}
```

Root cause
----------

ReportAOTTable's per-state action loop at lime.c:5832-5854 only
emits explicit cases for SHIFT / SHIFTREDUCE / REDUCE action
types:

```c
for(ap=stp->ap; ap; ap=ap->next){
    if( ap->type==SHIFT ){ ... }
    else if( ap->type==SHIFTREDUCE ){ ... }
    else if( ap->type==REDUCE ){ ... }
    /* falls through silently for other types */
}
```

But `enum e_action` (lime.c:185) has more cases:

  SHIFT, ACCEPT, REDUCE, ERROR,
  SSCONFLICT, SRCONFLICT, RRCONFLICT,
  SH_RESOLVED, RD_RESOLVED, NOT_USED, SHIFTREDUCE

The table-driven path encodes ERROR action types as
YY_ERROR_ACTION entries in yy_action[] when yy_lookahead[i]
matches the lookahead.  AOT skips these in the emit loop, so
they fall through to the state's default reduce.

ACCEPT is similar.  `enum e_action` ACCEPT becomes
YY_ACCEPT_ACTION in yy_action[]; AOT silently drops it, so
final-state reductions fall through to the state default.

For PG's gram.lime we hit ERROR overrides everywhere; ACCEPT
likely too at the start state, but ERROR is the visible bug.


Fix sketch
----------

```c
/* lime.c:5832, ReportAOTTable per-state action loop.
 * Add explicit cases for ERROR and ACCEPT action types.
 */
for(ap=stp->ap; ap; ap=ap->next){
    if( ap->type==SHIFT ){ ... }            /* unchanged */
    else if( ap->type==SHIFTREDUCE ){ ... } /* unchanged */
    else if( ap->type==REDUCE ){ ... }      /* unchanged */
    else if( ap->type==ERROR ){
        fprintf(out, "        case %d: return %d; "
                "/* %s -> error action (explicit override) */\n",
                ap->sp->index,
                lemp->errAction,
                ap->sp->name);
    }
    else if( ap->type==ACCEPT ){
        fprintf(out, "        case %d: return %d; "
                "/* %s -> accept action */\n",
                ap->sp->index,
                lemp->acceptAction,
                ap->sp->name);
    }
    /* SSCONFLICT, SRCONFLICT, RRCONFLICT, SH_RESOLVED,
     * RD_RESOLVED, NOT_USED are presumably already collapsed
     * by CompressTables before ReportAOTTable runs.  Worth
     * a comment / sanity assert. */
}
```

The has_actions check at lime.c:5825 should also count ERROR
and ACCEPT actions as "actions worth emitting":

```c
for(ap=stp->ap; ap; ap=ap->next){
    if( ap->type==SHIFT || ap->type==SHIFTREDUCE
        || ap->type==REDUCE || ap->type==ERROR
        || ap->type==ACCEPT ){
        has_actions = 1;
        break;
    }
}
```

(Currently only SHIFT/SHIFTREDUCE; if a state has only ERROR
overrides + a default reduce, has_actions stays 0 and the
emit takes the no-actions branch which only emits the default
return.  States that mix SHIFT with ERROR overrides currently
work for SHIFT but lose ERROR -- exactly what we observed.)
