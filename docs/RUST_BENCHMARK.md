# Lime Rust Output vs the Rust Parsing Ecosystem

This document compares Lime's `--rust` + `--rustlex` outputs
against the most-used Rust-native parser/lexer crates on a
common JSON parsing benchmark.

## Setup

- **Machine:** Intel i9-12900H (24 thread, 4.7 GHz turbo), Linux, GCC 15.2 / rustc 1.95-nightly via Nix dev shell.
- **Fixture:** `bench/rust_compare/fixture.json` -- 226,620 bytes (~227 KB). Generated by `make_fixture.py` with seed=42; mixed nested objects, arrays, strings, numbers, booleans, nulls.
- **Method:** Each benchmark is a separate binary (`bench/rust_compare/src/bin/bench_*.rs`) that:
  1. Reads the fixture.
  2. Warms up (10 iterations).
  3. Measures N timed iterations of the operation under test.
  4. Reports `us/parse` and `MB/s` derived from `fixture_bytes * iter / elapsed`.
- All competitors compiled with `--release`, `lto = "fat"`, `codegen-units = 1`.
- All parsers do the **same work**: parse the JSON to "OK or error" -- no AST construction except where the API forces it (serde_json builds a Value).

|---|---|---:|---:|---:|

## What changed in v0.8.2

The `--rustlex` emit was rewritten to a flat-dense layout with
a row-pointer hoist:

| Aspect | v0.8.1 | v0.8.2 |
|---|---|---|
| Transition table layout | `&[[i16; 256]]` (nested) | `&[i16]` (flat, indexed `state*256+b`) |
| Inner-loop dispatch | `trans[state][b]` (nested-array addressing) | row pointer + `*row.add(b)` |
| LOC of generated lexer | 704 (json) | 709 (json -- comment-heavy) |
| Memory per state group | 16 KB (32 states * 256 * i16) | 16 KB (unchanged) |
| Tokenize throughput | ~225 MB/s | ~265 MB/s (**+18 percent**) |
| Gap vs logos | 1.88x | **1.74x** |

The row-pointer hoist lets the compiler keep `row` in a
register across the byte-loop body instead of recomputing
`state * 256` on every iteration.  LLVM's nested-slice
addressing in v0.8.1 was apparently not seeing this
optimisation through `[[i16; 256]]`.

## What we tried and rejected

**Equivalence-class compression** (logos's secret sauce for
small DFAs): two bytes share a class iff they have identical
transitions across every state.  For the JSON DFA this
collapses 256 bytes to 26 classes -- table size drops 8x.

We implemented it, benchmarked it, and **it regressed perf
by 28 percent**:

| Layout | LOC | Memory | Tokenize MB/s |
|---|---|---|---|
| v0.8.1 nested dense | 704 | 16 KB | 225 |
| v0.8.2 flat dense   | 709 | 16 KB | **265** |
| class-compressed    | 312 | 2 KB  | 200 (regression) |

The class layout adds a dependent load (`byte_class[b]`
before `trans[state*nclasses+class]`) that defeats cache
locality.  For small DFAs the dense table fits in L1 cache;
the extra indirection costs more than the cache-pressure
relief.  Class compression would win for very large DFAs
(1000+ states where dense doesn't fit in L1) -- deferred
until a consumer with a large grammar needs it.

Source code in `src/lex/emit_rust_lex.c` includes a comment
documenting this experiment so a future reader doesn't make
the same wrong inference from the structural simplicity of
the class layout.

## What would actually close the gap to logos

logos hits ~460 MB/s on this fixture; lime --rustlex hits
~265 MB/s.  The remaining 1.74x is not closable by tweaking
the table representation.  Real options:

1. **Per-token DFA emission** -- one DFA per token kind
   instead of one unified DFA.  Tokens that don't match
   early are abandoned without consuming the rest of the
   character stream.  ~500-1000 LOC of new emit code.
2. **SIMD whitespace skipping** -- platform-specific (x86
   PSHUFB, ARM NEON).  Roughly 200 LOC plus runtime
   feature detection.  Would close most of the gap on
   JSON-shaped inputs (~30 percent of bytes are skipped
   whitespace).
3. **Inlined match-on-(state, byte)** -- emit the DFA as a
   giant `match` statement instead of a table.  LLVM
   compiles small-arity matches to jump tables that
   sometimes outperform table loads.  Risk: emit blow-up
   on large DFAs; needs a heuristic.

None of these are landed in v0.8.2.  All are listed in
the roadmap for follow-up if a consumer's profile shows
the lexer as the bottleneck.

## Interpretation

### Lexer (--rustlex vs logos)

logos is **1.74x faster** than lime's lexer (460 vs 265 MB/s).
Down from 1.88x in v0.8.1 thanks to the flat-layout / row-
hoist rewrite.

logos's per-token DFA emission and SIMD-accelerated character
classification are the remaining wins.  Closing further
requires the work in the previous section.

### Parser (--rust vs lalrpop)

**lime parse: 660 us (327 MB/s)**
**lalrpop parse: 425 us (509 MB/s)**

lalrpop is **1.55x faster** than lime on parse-only.  Same
ratio as v0.8.1 (parse code wasn't touched in v0.8.2).
lalrpop emits direct state-machine code: one big match arm
per state with inlined goto logic, no indirect calls, no
per-reduce allocation.

Closing the parser gap would mean giving up runtime
composition (lime's main differentiator).  Acceptable for
parsers that want runtime composition or grammar mutation;
not the right tool for pure-static maximum-speed parsing.

### Combined (lex + parse)

**lime: 1577 us (137 MB/s)**
**lalrpop+logos: 776 us (278 MB/s; 2.03x faster)**

The combined gap is the product of both individual gaps.

### Where lime wins or ties

- **vs pest (PEG):** lime is **1.16x faster** combined.
  PEG backtracking is heavier than LALR on this fixture.
- **vs serde_json (full AST):** lime is **1.34x faster**
  combined.  serde_json's AST allocation is heavier than
  lime's table-driven LALR token-stream consumption.
- **vs nom/chumsky:** nom is faster combined (2.97x) but
  doesn't have a separate lexer phase; the recursive-descent
  design avoids token-stream allocation entirely.  Apples
  to oranges with lime's table-driven LALR.

## What this means for Lime's positioning

Lime's Rust output is **production-viable for grammars where**:
- Runtime composition is needed (lime's main differentiator;
  lalrpop has no equivalent).
- The C side is the primary target and Rust is additive
  (PG-team's pattern).
- Speed is "good enough" -- within 2x of the fastest
  specialized tools.

Lime's Rust output is **not the best choice when**:
- Pure static parser speed is the only concern (use
  lalrpop or hand-tuned recursive descent).
- The lexer alone needs to hit logos's 400+ MB/s (use
  logos directly or wait for lime's per-token DFA work).
- Building a JSON-style AST is the goal (use serde_json).

## Reproducing

```bash
cd bench/rust_compare
python3 make_fixture.py
cargo build --release
for b in serde lime logos lalrpop nom pest; do
    echo "=== bench_$b ==="
    ./target/release/bench_$b
done
```

All numbers above are median of 5 runs on a quiet
i9-12900H.  Run-to-run variance is ~5 percent.  Numbers will
differ on other hardware; the relative ordering should hold.

## Notes on individual tools

- **logos:** v0.14. Generates per-token state machines via proc-macro. Industry-standard fastest Rust lexer for hand-written-like throughput.
- **lalrpop:** v0.20. Rust-native LALR(1) parser generator. Direct analog to lime's parser side.
- **nom:** v7. Parser combinator library; combined lex+parse via recursive descent, no separate token stream.
- **pest:** v2. PEG parser generator. Slower than LALR for this fixture due to PEG's backtracking semantics.
- **chumsky:** v0.10. Modern parser combinator library. Skipped in this comparison; v0.10's API is substantially rewritten and a fair port is a separate exercise. nom serves as the parser-combinator data point.
- **serde_json:** v1. Industry-standard JSON parser; tuned for AST construction. The "speed of light" reference for general-purpose JSON parsing in Rust.

## Conclusions

1. **Lime's Rust output works correctly on a 226 KB JSON fixture with 69k tokens.** No correctness regressions.
2. **v0.8.2 closes ~7 percent of the gap to logos** via a flat-layout + row-hoist rewrite of the lexer emit.  No memory regression; ~18 percent faster tokenize.
3. **Performance is within 2x of the fastest specialized Rust tools** (logos, lalrpop) for the lexer and parser respectively, and roughly equal-or-better than higher-level tools (pest, serde_json's AST build, chumsky's API).
4. **The remaining 1.74x lexer gap to logos requires per-token DFA emission**, deferred to a future release pending consumer demand.
5. **For consumers whose needs are speed-of-light Rust JSON parsing, lalrpop+logos is the better choice.** For consumers who want grammar composition + Rust output + C output from one source, Lime is the only option in the ecosystem.
