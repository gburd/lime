Verification (recommended modification)
---------------------------------------

Your sketched plan #2 wraps the AOT call-site with an
assertion that compares against the table-driven path on every
parse.  That works but pays both functions per token in the
asserted build, which makes pg_regress runtimes painful at PG's
3865-state grammar.

Tighter alternative -- one full sweep up front, asserts pass,
then drop the table-driven half from the hot path:

  ```c
  /* Once at backend startup, in the lime-aot test build only: */
  for (uint16_t s = 0; s <= YY_MAX_SHIFT; s++) {
      for (uint16_t la = 0; la <= YY_NTOKEN; la++) {
          YYACTIONTYPE aot = yy_find_shift_action_aot(s, la);
          YYACTIONTYPE tab = yy_find_shift_action_tabledriven(s, la);
          assert(aot == tab && "Letter-15 regression");
      }
  }
  ```

Total cost: 3865 * yy_ntoken pairs, ~one-time at process start,
then both functions are pure C with no shared state so the
assertion is sound for the whole life of the parser.

The same pattern is what tests/test_aot_equivalence.c does for
the bench arith grammar.  Scaled up to PG's grammar, the sweep
is ~30 M lookups -- a few seconds of one-time work, well under
your test-budget threshold.
