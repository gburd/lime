Bug report: lime -j crash
-------------------------

Reproduced on every grammar tested (calc, datalog, syncrep_gram,
gram).  Root cause was a libc / arena allocator mismatch in
`ReportAOTTable` (lime.c:5775):

```c
z = file_makename(lemp, "_aot.c");   /* allocated via lime_malloc */
out = fopen(z, "wb");
if (out == 0) {
    fprintf(stderr, "Can't open file \"%s\".\n", z);
    free(z);                          /* WRONG: lime_free */
    return;
}
free(z);                              /* WRONG: lime_free */
```

`file_makename` allocates via the lime arena allocator, not the
libc heap.  jemalloc and glibc both crashed on the mismatched
free.  Fixed in commit `b566478`.

Verified end-to-end on calc.lime and datalog.lime: the resulting
`*_aot.c` is valid C (compiles cleanly with -O2 -Wall -Wextra)
and produces a working `yy_find_shift_action_aot` switch table.
The bug is closed.


API rename: lemon_* -> lime_*
-----------------------------

Eight public functions renamed in v0.2.5:

    lemon_snapshot_create         -> lime_snapshot_create
    lemon_snapshot_acquire        -> lime_snapshot_acquire
    lemon_snapshot_release        -> lime_snapshot_release
    lemon_snapshot_extend         -> lime_snapshot_extend
    lemon_modifications_to_grammar_text
                                  -> lime_modifications_to_grammar_text
    lemon_extension_registry_init -> lime_extension_registry_init
    lemon_extension_registry_destroy
                                  -> lime_extension_registry_destroy
    lemon_parser_version          -> lime_parser_version

include/parser.h has a backward-compatibility shim block so your
existing code compiles unchanged:

```c
#define lemon_snapshot_create   lime_snapshot_create
#define lemon_snapshot_acquire  lime_snapshot_acquire
#define lemon_snapshot_release  lime_snapshot_release
... (etc)
```

Macro-level (not function-wrapper) so the deprecated calls have
zero runtime cost and your debug symbols still point at the real
function.  We're slating removal of the shim for v0.4.0 -- plenty
of headroom for your migration.


Q1 / Q2 answers (excerpts from docs/INTEGRATING_LIME.md)
--------------------------------------------------------

The full doc is at docs/INTEGRATING_LIME.md in v0.2.5.  Three
direct answers below; consult the doc for code samples.

### Q1.1 -- Calling static reduce actions from parse_token

Today's runtime engine does not invoke reduce-action callbacks
at parse time.  The action tables in the snapshot encode the LR
machine; running the user's `{ ... }` body is reserved for a
future engine extension.  PG's existing static-parser path
(Shape A in the integration doc) is the right home for action
callbacks today.

When the runtime engine grows reduce-callback dispatch (tracked
in docs/ROADMAP.md), the integration shape will be: a single
opaque user-data pointer threaded through parse_begin, with the
engine calling a per-rule callback at each reduction.  The PG
team can prototype against the existing snapshot machinery
today and the callback wiring will light up transparently.

### Q1.2 -- parse_token's int location parameter

  - `location` is treated as the byte offset (or column) of
    the token's first character.  Pass `-1` if you don't have
    one available.
  - It maps to the `yyloc` slot in the engine's stack entry for
    that token.  When `%locations` is set in the grammar, user
    actions can read it via the same `@N` syntax Bison uses.
  - For PG's `YYLTYPE { int first_column; int last_column; }`
    convention, pass `loc.first_column`.  The `last_column`
    half is currently unused by Lime (PG's own usage of it has
    trailed off since 11 anyway).

### Q1.3 -- ParseContext lifecycle vs Bison's per-parse

Yes, parse_begin / parse_end are cheap -- they are pool
allocations from the snapshot's arena, not full process-level
state.  A single backend that creates one MemoryContext per
parse can call parse_begin inside that context and parse_end at
MemoryContext reset.  We expect to land a parse_reset(ctx)
helper in v0.3.0 that recycles the ParseEngine without releasing
the snapshot reference; until then, parse_end followed by a new
parse_begin is the right shape.

### Q2.1 -- JIT and the static action-table dispatch

The JIT operates on a ParserSnapshot.  It does NOT directly
accelerate the static yy_find_shift_action in your generated
gram.c.  If you want JIT acceleration, you need the snapshot
path (the ParserSnapshot + parse_token shape -- "Shape B" or
"Hybrid" in the integration doc).

### Q2.2 -- AOT (-j flag) as the static-codegen equivalent

Yes.  `lime -j grammar.lime` emits a `grammar_aot.c` containing

```c
YYACTIONTYPE_AOT yy_find_shift_action_aot(stateno, lookahead);
```

implemented as a switch-of-switches that the C compiler optimises
into a jump table at -O2.  That's the static-time equivalent of
the JIT: no runtime LLVM, no per-grammar compile delay, but the
same constant-folded dispatch shape.  Compile the AOT file
alongside your gram.c, link both, define `-DYYAOT` so the
generator's yy_find_shift_action forwards to the `_aot` version.

The bug from your Letter 14 (`free(): invalid size`) is fixed
in v0.2.5; lime -j now produces a valid `*_aot.c` on every
grammar tested.


Hybrid path (probably your deployment shape)
--------------------------------------------

For a backend like PG that wants per-process throughput on the
common case but the option to load dialect extensions at runtime,
the hybrid shape (full description in the integration doc):

  1. Build the production parser with the static path (Shape A
     in the doc): static gram.c emitted by lime, action callbacks
     compiled in, base_yyparse invoked from raw_parser.  This is
     the hot path; no runtime engine overhead.
  2. Provide a parallel ParserSnapshot built via
     lime_snapshot_create that **shares the same grammar source**.
     When an extension is loaded, dispatch routes through the
     snapshot path; when no extensions are loaded, the static
     path is used.

Headline performance number on the static path: 1.04-1.81×
faster than Bison across architectures (full table in
docs/BENCHMARKS_VS_BISON.md).  Headline on the snapshot path:
JIT typically gives 2-3× over the C interpreter on grammars
> 100 states on aarch64; on x86 the JIT is auto-skipped for
small grammars because the unrolled-switch interpreter is
already optimal for that size class.


Adoption ledger
---------------

Closed in v0.2.5:

  - lime -j AOT crash (your blocking bug from Letter 14)
  - lemon_* -> lime_* public API rename (with compat shim so
    your existing code keeps working unchanged)
  - dlopen test failures on FreeBSD (test_snapshot_create is

    + actions (docs/INTEGRATING_LIME.md, 225 lines)

Open (architectural; pending):

  - Reduce-callback dispatch in the runtime engine.  Required
    for full Shape B adoption (currently you'd lose action
    bodies on the snapshot path).  Roadmap item.
  - parse_reset helper to recycle ParseEngine without snapshot
    re-acquire.  Targeting v0.3.0.

Not in this letter:

  - Win11 aarch64 + MSVC testing (santorini).  Tried; SSH was
    not responsive.  Deferred to next round.  We'll need to
    deal with weak-symbol portability on Windows when we do
    (Windows / MSVC has neither `__attribute__((weak))` nor
    `--unresolved-symbols=ignore-all`).  Likely path: the .so
    case on Windows links snapshot.c into the .dll directly
    (which is what we tried first on FreeBSD before falling
    back to the linker-flag approach).
